00. Hallazgo principal

La API pública de resultados electorales de la ONPE es accesible sin autenticación, sin tokens, sin rate limiting. No por descuido técnico. Por diseño de presentación: el sistema fue construido para mostrar datos a un navegador, y cualquier script que imite ese comportamiento obtiene exactamente lo mismo.

10 endpoints mapeados
0 endpoints con auth
0 rate limits detectados

El mecanismo

CloudFront (el CDN de Amazon que sirve el sitio) está configurado con comportamiento dual basado en el header Sec-Fetch-Dest:

http # Navegación directa (curl sin headers especiales, browser en barra de URL) Sec-Fetch-Dest: document → CloudFront devuelve index.html (SPA Angular, 35KB) # Request XHR/fetch desde JavaScript (browser, o script que imita) Sec-Fetch-Dest: empty Sec-Fetch-Mode: cors Sec-Fetch-Site: same-origin → CloudFront proxea al backend real → JSON con datos electorales

El script onpe_recon.py reproduce exactamente estos headers. La API responde.

Base URL real del backend: resultadoelectoral.onpe.gob.pe/presentacion-backend/

Las URLs del estilo /resumen-general/proceso-electoral-activo que circulan en redes son rutas del router Angular, no endpoints de API. El SPA las intercepta. Por eso curl sin headers correctos devuelve HTML.

Lo que la API nunca expone

  • Timestamp de ingreso de cada acta al sistema central — imposible reconstruir el orden de procesamiento
  • ID único de mesa de votación — sin este campo, los votos son totales sin ubicación granular
  • Secuencia de procesamiento de actas — no hay rastro de qué actas llegaron primero, cuáles después
  • Metadatos de transmisión — sin dispositivo emisor, sin operador, sin hora de envío por acta
  • Snapshots históricos del conteo parcial — la API solo devuelve estado actual; no existe endpoint de historial
  • Distribución de votos nulos/blancos por mesa — solo agregados nacionales o por distrito
  • Actas por candidato en orden cronológico — la curva de escrutinio no existe como dato público

02. El endpoint ausente — y el que sí funciona

El endpoint /actas/listar existe en el backend. Responde. Y rechaza toda consulta sin los parámetros correctos.

http GET /presentacion-backend/actas/listar Accept: application/json Sec-Fetch-Dest: empty ← HTTP 400 { "error": "Solicitud inválida", "message": "Los parámetros enviados son incorrectos o están incompletos.", "status": 400 }

Se probaron combinaciones sistemáticas: idEleccion=10, idMesa, proceso, page, size, idDistritoElectoral, idProceso. Todas devuelven 400. Los parámetros correctos no están documentados públicamente.

Si /actas/listar expusiera datos completos, permitiría construir:

— La curva de escrutinio: votos(candidato, t) para cada acta ingresada
— Detección de late-count bias estadístico
— Trazabilidad de qué mesas llegaron tarde y cómo se distribuyeron sus votos

Ese endpoint existe. No es público. No está documentado. Es el único que haría transparente lo que este sistema dice ser.

02-B. El endpoint que sí funciona

Después de interceptar tráfico del SPA durante el escrutinio se encontró el endpoint granular real:

http GET /presentacion-backend/actas/buscar/mesa?codigoMesa=000001 Accept: application/json Sec-Fetch-Dest: empty ← HTTP 200 — 240KB { "success": true, "data": [ { "idMesa": 1, "codigoMesa": "000001", "idEleccion": 10, "nombreLocalVotacion": "IE 18288 ISABEL LYNCH DE RUBIO", "totalElectoresHabiles": 230, "totalVotosValidos": 134, "descripcionEstadoActa": "Contabilizada", "detalle": [ /* votos por candidato */ ], "lineaTiempo": [ { "codigoEstadoActa": "T", "descripcionEstadoActa": "Digitalización", "fechaRegistro": 1776065162000 }, { "codigoEstadoActa": "D", "descripcionEstadoActa": "Digitación", "fechaRegistro": 1776065284000 }, { "codigoEstadoActa": "C", "descripcionEstadoActa": "Contabilizada", "fechaRegistro": 1776065346000 } ] } ] }

Cada respuesta incluye 5 actas por mesa (una por tipo de elección). El campo lineaTiempo registra cada transición de estado con timestamp Unix. Los estados documentados:

CódigoEstadoDescripción
TDigitalizaciónActa física escaneada
DDigitaciónDatos ingresados manualmente
CContabilizadaValidada — entra al cómputo oficial
HObservadaImpugnada — fuera del cómputo
EPara envío al JEEDerivada a corte electoral

Limitación real: no existe un endpoint que devuelva todas las mesas en bloque. El acceso es por código de mesa individual. Auditoría completa = 92,501 peticiones separadas.

Estrategia alternativa con datos públicos:

— Muestreo estadístico: N mesas aleatorias → distribución de tiempos de pipeline → detección de outliers
— Polling de totales: snapshots cada N minutos de /totales → serie temporal artificial
— Correlación geográfica: ubigeo por mesa → concentración de anomalías por zona

Granularidad real existe. No es cómoda. Requiere trabajo.

Corrección de versión anterior: el primer análisis declaró que la API no expone timestamps ni IDs de mesa individuales. Eso era incorrecto. El endpoint /actas/buscar/mesa sí los expone. La tabla corregida:

La arquitectura está diseñada para presentar resultados, no para auditarlos.
— La granularidad existe. El acceso cómodo, no.

La ausencia de un endpoint bulk no implica ausencia de datos. Implica que la auditoría completa requiere 92,501 peticiones individuales. Técnicamente posible. No diseñado para facilitarlo.

Lo que SÍ permite esta API (versión corregida):

— Timestamps de pipeline por acta individual (Digitalización → Digitación → Contabilizada)
— Votos por candidato desagregados por mesa
— Estado legal de cada acta (normal / observada / en JEE)
— Muestreo estadístico: distribución de tiempos de procesamiento
— Detección de re-digitaciones (actas ingresadas más de una vez)
— Geolocalización de anomalías por ubigeo

La discrepancia entre presidencial y Senado DEU es estructural. El Senado es una elección nueva — primera cámara alta desde 1993. Los operadores no tienen procedimiento consolidado. Las 9 re-digitaciones en una sola acta son el extremo de ese problema, no evidencia de manipulación.

Hallazgo específico — mesas en el extranjero:

Los códigos de mesa >088000 devuelven HTTP 400 o no responden al endpoint estándar. La mesa 087619 (IFEMA MADRID) tiene pipeline de 58.9 horas — procesamiento diferido de votos en el exterior. El bloque de extranjero (~4,000 mesas) funciona con sistema distinto y es parte del cómputo "congelado" del presidencial.
