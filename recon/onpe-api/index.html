00. Hallazgo principal

La API pública de resultados electorales de la ONPE es accesible sin autenticación, sin tokens, sin rate limiting. No por descuido técnico. Por diseño de presentación: el sistema fue construido para mostrar datos a un navegador, y cualquier script que imite ese comportamiento obtiene exactamente lo mismo.

10 endpoints mapeados
0 endpoints con auth
0 rate limits detectados

El mecanismo

CloudFront (el CDN de Amazon que sirve el sitio) está configurado con comportamiento dual basado en el header Sec-Fetch-Dest:

http # Navegación directa (curl sin headers especiales, browser en barra de URL) Sec-Fetch-Dest: document → CloudFront devuelve index.html (SPA Angular, 35KB) # Request XHR/fetch desde JavaScript (browser, o script que imita) Sec-Fetch-Dest: empty Sec-Fetch-Mode: cors Sec-Fetch-Site: same-origin → CloudFront proxea al backend real → JSON con datos electorales

El script onpe_recon.py reproduce exactamente estos headers. La API responde.

Base URL real del backend: resultadoelectoral.onpe.gob.pe/presentacion-backend/

Las URLs del estilo /resumen-general/proceso-electoral-activo que circulan en redes son rutas del router Angular, no endpoints de API. El SPA las intercepta. Por eso curl sin headers correctos devuelve HTML.

Lo que la API nunca expone

  • Timestamp de ingreso de cada acta al sistema central — imposible reconstruir el orden de procesamiento
  • ID único de mesa de votación — sin este campo, los votos son totales sin ubicación granular
  • Secuencia de procesamiento de actas — no hay rastro de qué actas llegaron primero, cuáles después
  • Metadatos de transmisión — sin dispositivo emisor, sin operador, sin hora de envío por acta
  • Snapshots históricos del conteo parcial — la API solo devuelve estado actual; no existe endpoint de historial
  • Distribución de votos nulos/blancos por mesa — solo agregados nacionales o por distrito
  • Actas por candidato en orden cronológico — la curva de escrutinio no existe como dato público

02. El endpoint ausente

El único endpoint con potencial de auditoría real es /actas/listar. Existe en el backend. Responde. Y rechaza toda consulta sin los parámetros correctos.

http GET /presentacion-backend/actas/listar Accept: application/json Sec-Fetch-Dest: empty ← HTTP 400 { "error": "Solicitud inválida", "message": "Los parámetros enviados son incorrectos o están incompletos.", "status": 400 }

Se probaron combinaciones sistemáticas: idEleccion=10, idMesa, proceso, page, size, idDistritoElectoral, idProceso. Todas devuelven 400. Los parámetros correctos no están documentados públicamente.

Si /actas/listar expusiera datos completos, permitiría construir:

— La curva de escrutinio: votos(candidato, t) para cada acta ingresada
— Detección de late-count bias estadístico
— Trazabilidad de qué mesas llegaron tarde y cómo se distribuyeron sus votos

Ese endpoint existe. No es público. No está documentado. Es el único que haría transparente lo que este sistema dice ser.

Estrategia alternativa con datos públicos

Sin /actas/listar, la única forma de construir una serie temporal es polling agresivo: llamar /totales y /participantes cada N minutos y guardar cada snapshot. El reporte base del script onpe_recon.py es la infraestructura para ese approach.

bash # Polling cada 5 minutos durante el escrutinio activo while true; do python onpe_recon.py sleep 300 done # Resultado: serie temporal artificial de ~288 snapshots por día # Permite detectar: saltos anómalos, cambios de tendencia, inconsistencias # Limitación: granularidad de 5min, no de acta individual

03. Veredicto técnico

¿Es posible detectar beneficio
sistemático con estos datos?

NO — con los datos públicamente disponibles en la API de la ONPE.

Para detectar late-count bias o manipulación escalonada se requiere, como mínimo:

Requisito¿Disponible?
Timestamps de ingreso de cada acta❌ No expuesto
ID único por mesa de votación❌ No expuesto
Snapshots temporales del conteo parcial❌ No expuesto
Datos georreferenciados por mesa❌ No expuesto
Actas individuales con candidato + hora + mesa❌ No expuesto
Totales desagregados por candidato en tiempo real⚠️ Solo agregados finales sin historial
Progresión del conteo por distrito en el tiempo❌ No expuesto
La arquitectura está diseñada para presentar resultados, no para auditarlos.

Esto no es acusación. Es descripción técnica. La ausencia de trazabilidad no implica fraude. Implica que si el fraude existiera, no podría ser detectado con las herramientas públicas disponibles. Son cosas distintas que se confunden consistentemente en el debate público.

Lo que SÍ permite esta API:

— Verificar totales finales por candidato (snapshot actual)
— Observar % de actas contabilizadas y pendientes de envío al JEE
— Comparar cifras con actas físicas disponibles en portales oficiales
— Desagregar resultados por distrito electoral (sin timestamps)
— Construir un monitor de avance de escrutinio si se hace polling

Renovación Popular — el partido de Rafael López Aliaga, tercer lugar en la presidencial — lidera el Congreso en Lima Metropolitana con 19.99%, usando el mismo sistema de escrutinio, procesado el mismo día, transmitido por la misma infraestructura técnica.

El candidato que denuncia irregularidades en la elección presidencial obtiene su mejor resultado en la elección legislativa limeña a través de ese mismo sistema.

Esto no prueba ni refuta nada sobre el escrutinio presidencial. Es un dato que raramente aparece en el debate sobre la integridad de esta elección.

La consistencia técnica entre ambas elecciones — mismo sistema, mismo día, misma API — no valida ni invalida ninguna denuncia específica. Lo que sí hace es complicar la narrativa de un sistema selectivamente corrupto. Si el mecanismo beneficia a alguien en presidencial, opera el mismo día para perjudicarlo en legislativo limeño, y viceversa.
